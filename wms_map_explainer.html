A visual guide to understanding warehouse map concepts, how the map improves picking wave efficiency, how to draw a map in WMS, and what Brazil needs to do next.

1

WMS Map — Core Concepts

The WMS 2D Warehouse Map is a digital twin of the physical warehouse floor. It encodes exactly where shelves, aisles, walls, and workstations are — so the system can calculate how far a picker actually has to walk between any two storage locations.

The 3-level location hierarchy

Zone (储区)

A named area within the warehouse, e.g. Normal Storage, High Value, Staging. Defines activity type, storage type, and picking rules.

Pathway (通道)

An aisle inside a Zone. Configured with number of segments × columns × floors. Inherits Zone attributes.

Cell / Location (储位)

An individual bin. Code = WH + Zone + Pathway + Segment + Floor + Column. This is where stock is physically stored.

Figure 1 — Top-down view of a warehouse: Zones contain Pathways; Pathways contain Cells (individual bins).

What the map encodes

📐 Physical layout

Exact positions of racks, walls, pillars, and open aisles drawn as elements on the 2D canvas.

🔗 Location binding

Each storage cell in WMS is "bound" to a pixel coordinate on the map so the system knows where it physically is.

🏁 Start & end points

Markers that define where pickers enter the floor and where they drop off picked goods (sorting / checking station).

Key rule: A location that is not bound to the map is invisible to the routing engine. The system cannot generate a route for it, and it will be excluded from wave optimisation.
2

How the Map Improves Wave Efficiency

A picking wave is a batch of orders grouped together so one picker (or a few pickers) can collect all items in a single floor sweep. The key optimisation question is: which orders should be batched together, and in what sequence should the picker visit each location?

Without a map — location weightage score (current BR approach)

Without a map, the system estimates the "distance" between two locations using a weightage score formula based on location ID components:

// Score between Location A and Location B
Score = (Zone_A − Zone_B) × 100
      + (Pathway_A − Pathway_B) × 10
      + (Segment_A − Segment_B) × 1

The system then groups wave locations to minimise the total score — treating a lower score as a shorter distance. This works reasonably well when location IDs are assigned in a layout-consistent way, but breaks down in two common real-world situations:

⚠ Problem 1 — Nearby locations labeled as different zones

Two locations physically side-by-side (e.g. back-to-back racks facing opposite aisles) may belong to Zone 01 and Zone 05. The score difference is (5−1)×100 = 400 — the system treats them as far apart, when in reality the picker only takes a single step.

⚠ Problem 2 — Cross-aisle shortcuts not captured

Many warehouses have a walk aisle in the middle of a pathway that cuts across multiple rack rows. Two locations in different pathways may be reachable via this short cross-aisle — their actual walking distance is much smaller than the score suggests. The formula has no concept of aisles, shortcuts, or walkable paths; it only sees numbers.

Result: pickers are routed on a sub-optimal sequence. Wave batches locations that look close by ID but are physically far, and misses genuine shortcuts — leading to unnecessary back-and-forth walking.

What map-based routing looks like in practice

Once a warehouse map is set up, the system can display and track the exact numbered picking sequence for every sub-picking task — stop by stop, through the real walkable aisles of the warehouse. Operations teams can verify routes, troubleshoot inefficiencies, and confirm that the picker followed the optimal path.

With a map — shortest-path distance (A* algorithm)

The map is rasterised into a grid. Every pair of picking locations has a pre-computed actual walkable distance (stored in Hive). When wave generation runs, it uses these real distances to:

  1. Group orders whose items are physically close together into the same wave
  2. Sequence locations within each wave in walking order (S-path / U-path)
  3. Minimise total walking distance per wave, directly reducing pick time

End-to-end flow

  • 1
    Map drawn & published in WMS

    Ops team draws racks, aisles, and markers in the map editor. Each location is bound to its pixel coordinate on the map.

  • 2
    Distance pre-computation (A* / JPS)

    Offline: the routing engine rasterises the map into a grid and runs pathfinding for every pair of picking locations. Results are stored in Hive table ci_wms.wm_pathing_route.

  • 3
    Wave generation uses real distances

    When wave runs, it looks up pre-computed distances to group and sequence locations. Locations without distance data are excluded.

  • 4
    Picker follows optimised task sequence

    PDA shows the pick list in walking order. Shorter total walking distance → more orders picked per hour (UPH ↑).

How to Draw a Warehouse Map in WMS

The map is drawn in the WMS Map Editor — a 2D canvas where you place elements representing the physical warehouse. Here is the step-by-step process.

📖 Official User Manual: WMS Warehouse Map — Step-by-step Drawing Guide (Google Slides)

Before you start — what you need

📄 Floor plan

A layout drawing or PDF of the warehouse floor showing rack positions, aisle widths, walls, and workstation locations.

📋 Storage Area Setting

All Zones, Pathways, and Cells must already be configured in WMS before binding. The map links to existing location records.

🏁 End-point list

Know where pickers start and where they drop goods (sorting station, checking station, packing area, etc.).

Drawing steps

  • 1
    Open Map Editor → Create new map for the warehouse

    Navigate to WMS → Warehouse Map → Map File List. Click Create, select the warehouse (WH_ID). One map per warehouse.

  • 2
    Set the canvas scale

    Configure the grid precision (metres per grid cell). Finer precision = more accurate routes but heavier computation. Recommended: match your narrowest aisle width.

  • 3
    Draw walls and building boundaries

    Use the Wall element to trace the outer walls and internal partitions. These become hard obstacles in the routing grid — the algorithm will never route through them.

  • 4
    Draw racks / shelves

    Place Rack elements for each shelf row. The system treats rack bodies as obstacles. The aisle in front of each rack face is the walkable path. Important: the system validates that no two elements overlap — if they do, a position conflict error is shown immediately.

  • 5
    Place pathway / aisle elements

    Mark the main aisles (cross-aisles) as Path elements. These confirm walkable corridors and help the A* grid correctly identify passable nodes.

  • 6
    Set Start Point and End Point markers Required

    Place at least one End Point marker at the workstation/sorting area where pickers deliver goods. You can configure multiple end points by station type: Checking / Sorting / Sorting While Checking / Packing. Without an end point, the map cannot be saved or published.

  • 7
    Bind storage locations to map positions

    In the Location Binding panel, select each WMS location (cell code) and click its physical position on the map. The system auto-suggests the nearest rack face. Binding rate is tracked: All Location Binding Rate and Picking Location Binding Rate are shown in the editor. Target: 100% picking location binding before enabling wave.

  • 8
    Save → Verify metrics → Publish

    After saving, check the All Routes Success Rate — this shows what % of location pairs have a valid walkable route. Investigate any blocked paths via Report Center → Warehouse Map → Blocked Picking Paths. When satisfied, click Publish to make the map live. The routing engine will then run distance pre-computation.

⚠ Don't skip binding! A published map with low binding rate is worse than no map — the wave engine will produce incomplete results and silently skip unbound locations.
4

Segment-level vs Column-level Distance Calculation

This is the most critical technical issue currently affecting WMS map routing accuracy — and the reason Brazil (BR) warehouses cannot yet use the map feature.

Understanding the location hierarchy inside a pathway

Within a single Pathway, locations are organised as:

Pathway → Segment (格) → Column (列) → Floor (层)

A Segment is a structural section of the rack (think: one bay frame). Each segment can contain multiple Columns side by side. In most ID/MY warehouses, 1 segment = a few columns. But in BR, 1 segment can contain 100–300 column IDs — meaning one "bay" spans a very long stretch of the aisle.

⚠ CURRENT: SEGMENT-LEVEL CALCULATION

Distance between two locations is calculated using the midpoint of their respective segments as the representative coordinate.

Problem: Loc A is at column 1 of segment 1, Loc B is at column 100 of segment 2. But the system calculates the distance between the two midpoints — which is far shorter than the real walking distance. The wave thinks these two locations are closer than they actually are.

✓ PROPOSED FIX: COLUMN-LEVEL CALCULATION

Distance is calculated using the actual column coordinate of each location — its true physical position on the shelf face.

Result: Loc A (col 1) and Loc B (col 200) now use their precise physical positions. The distance correctly reflects the full length of the aisle the picker must traverse. Wave grouping is accurate.

Why this matters especially for Brazil

Market Columns per Segment Impact of segment-level calc Map usable today?
ID / MY / SG Typically 3–10 cols/segment Small error — midpoint is close enough to real position ✓ Yes
BR 100–300 cols/segment Midpoint is far from actual position — error can be 50+ metres per location pair ✗ Blocked
TW High + encoding bug Same problem + column ID reuse across segments ✗ Blocked
The fix (in progress): The tech team is updating the routing engine to calculate distances at column level — using the precise column coordinate of each location instead of the segment midpoint. This fix is being done together with the JPS mega-warehouse performance fix (Issue #2, already deployed for MYK). Once released, BR warehouses with a drawn map will be able to use map-based wave optimisation immediately.
